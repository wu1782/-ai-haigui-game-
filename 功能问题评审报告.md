# AI 海龟汤项目功能问题评审报告（产品经理视角）

评审时间：2026-04-09  
评审范围：前后端核心流程（登录鉴权、AI 判定/生成、游戏主流程、结果页闭环）  
评审方式：基于代码与配置静态走查（未做完整人工回归）

---

## 一、结论摘要

项目整体功能设计完整，覆盖了“玩、社交、成长、内容生产”的闭环，产品方向是对的。  
但当前存在若干**高优先级功能一致性问题**，会直接影响：

1. AI 能力是否真的可用（判定/生成功能可出现“看似成功但实际无效”）
2. 账号安全链路是否可闭环（忘记密码在某些环境下可能不可用）
3. 游戏核心页面是否稳定（语音输入分支存在运行级风险）

如果不修复，线上会出现“用户能进入页面但关键功能体验异常”的情况。

---

## 二、问题清单（按优先级）

## P0（必须立即修复）

### 1) AI 判定前端响应解析字段与后端返回结构不一致
- **现象**：前端读取 `data.answer`，但后端实际返回 `data.data.answer`。
- **影响**：AI 判定可能频繁走降级或异常分支，用户会感觉回答不稳定甚至“随机”。
- **证据**：
  - 前端：`src/services/aiService.ts` 中 `getAIResponse` 返回 `answer: data.answer || '无关'`
  - 后端：`server/index.js` `/api/v1/ai/judge` 返回结构为 `{ success: true, data: { answer } }`
- **产品风险**：核心卖点“AI 裁判”可信度下降。
- **建议**：统一响应协议，前端改为解析 `data?.data?.answer`；同时保留兼容兜底并埋点监控解析失败率。

### 2) AI 生成故事前端解析字段同样不一致
- **现象**：前端读取 `data.title/surface/bottom...`，后端返回 `data.data.title/surface/bottom...`。
- **影响**：用户触发“AI 生成”后可能拿到空故事或默认占位内容。
- **证据**：
  - 前端：`src/services/aiService.ts` `generateStory`
  - 后端：`server/index.js` `/api/v1/ai/generate` 返回 `{ success: true, data: { ...story } }`
- **产品风险**：UGC/AI 供给链路失效，影响留存与传播。
- **建议**：前后端约定统一 API Contract（建议 OpenAPI + TS 类型共享），避免重复错位。

### 3) 忘记密码链路在未配置 JWT_SECRET 时存在失效风险
- **现象**：重置邮件 token 签发使用 `EFFECTIVE_JWT_SECRET`（有开发兜底），但校验时用 `JWT_SECRET`（可能为空）。
- **影响**：开发/测试环境中“邮件能发出但 token 无法验证”。
- **证据**：`server/services/emailService.js`
  - 签发：`jwt.sign(..., EFFECTIVE_JWT_SECRET, ...)`
  - 校验：`jwt.verify(token, JWT_SECRET)`
- **产品风险**：账号恢复闭环失败，投诉风险高。
- **建议**：校验统一使用 `EFFECTIVE_JWT_SECRET`，并在启动时输出配置健康提示。

## P1（应尽快修复）

### 4) 游戏页语音状态变量使用不完整，存在运行时错误风险
- **现象**：页面逻辑引用 `isListening`，但 `useVoice()` 解构中未取出该字段。
- **影响**：语音相关流程可能抛异常，影响游戏页面稳定性。
- **证据**：`src/pages/Game.tsx`
  - 解构：`const { transcript: voiceTranscript } = useVoice()`
  - 使用：`if (voiceTranscript && !isListening) { ... }`
- **产品风险**：核心流程中断，用户流失。
- **建议**：补齐解构并补语音开关的端到端冒烟用例。

### 5) 已验证邮箱用户仍可能被重复触发“发送验证邮件”
- **现象**：接口读取用户时未查 `emailVerified` 字段，却在后续判断中使用。
- **影响**：已验证用户仍可收到验证邮件，造成困扰与资源浪费。
- **证据**：`server/routes/auth.js` `/send-verify-email`
  - 查询：`SELECT id, username, email FROM users ...`
  - 判断：`if (user.emailVerified) ...`
- **产品风险**：通知体验差、运营成本增加。
- **建议**：查询补字段并加接口单测。

### 6) 文档与实际 API 版本路径不一致
- **现象**：README 中有 `/api/auth/*` 描述，但项目实际主路由为 `/api/v1/*`。
- **影响**：联调、新人接入和运维排障效率下降。
- **证据**：
  - 文档：`README.md`
  - 实际：`server/index.js` 使用 `API_PREFIX = /api/v1`
- **产品风险**：对外接入成本升高。
- **建议**：文档统一为 v1 路径，保留旧路径兼容说明。

## P2（优化项）

### 7) 仓库内出现重复/冲突文件迹象（路径风格不一致）
- **现象**：存在 `server/index.js` 与 `server\index.js` 等重复条目（同类情况在 auth/email 文件亦可见）。
- **影响**：多人协作时易引发覆盖、误读和发布差异。
- **建议**：统一路径与提交规范，清理重复文件并加 pre-commit 校验。

### 8) AI 服务降级策略对用户感知不友好
- **现象**：AI 失败时部分路径返回“随机模拟回答”或笼统报错。
- **影响**：用户不易判断系统异常，降低信任。
- **建议**：降级改为“明确提示 + 重试按钮 + 异常追踪ID”，禁用随机回答。
